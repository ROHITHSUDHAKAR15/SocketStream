# Security model

SocketStream is **end-to-end encrypted**: encryption keys are generated in the
browser and the server only ever sees public keys and ciphertext. This document
states precisely what that does and does not protect against — being honest about
the boundaries is the point.

## What the server knows

| Stored on the server | Never on the server |
|---|---|
| username | password (only a bcrypt **hash** of it) |
| bcrypt password hash | RSA **private** key |
| RSA **public** key | plaintext of any message |
| per-recipient **ciphertext** + timestamps | AES content keys |

The private key is generated by `crypto.subtle.generateKey` in the browser,
exported, encrypted with a key derived from your password
(PBKDF2-SHA256, 200k iterations → AES-256-GCM), and kept in `localStorage`.
It is **never transmitted**. The server has no endpoint that accepts or returns a
private key. The test suite pins this down
(`tests/test_server.py::test_db_never_stores_private_keys`).

## How a message is protected

Per message, the sender's browser:

1. generates a fresh **AES-256-GCM** content key and a random IV,
2. encrypts the plaintext with it,
3. wraps the AES key to the recipient's **RSA-2048-OAEP** public key,
4. concatenates `wrapped-key ‖ iv ‖ ciphertext`, base64-encodes it, and POSTs it.

A separate copy is encrypted to the **sender's own** public key so the sender can
read their own history — there is no plaintext self-copy. Broadcasts are encrypted
once per recipient. The server stores each blob verbatim and relays it; only the
holder of the matching private key can decrypt.

## Threats this defends against

- **A curious or breached server / database dump.** Messages and private keys are
  unreadable without a user's password-derived key, which the server never has.
- **Passive network eavesdropping.** TLS in transit, plus the payload is already
  ciphertext.
- **Stored XSS via message content.** Messages are rendered with `textContent`,
  never `innerHTML`; usernames are validated to `[A-Za-z0-9_]`.
- **Password theft from the DB.** Passwords are bcrypt-hashed, never stored.
- **Login brute force.** Per-IP rate limiting on `/login`.

## Honest limitations (by design, for a portfolio demo)

- **The browser is trusted.** After login the unwrapped private key lives in the
  tab's `sessionStorage`. A successful XSS or a malicious server-served script
  could read it. True hardening would isolate keys in a worker / WASM sandbox or a
  hardware token. The app ships a strict CSP-friendly structure (no inline message
  rendering) to reduce XSS surface, but does not yet send a CSP header.
- **No forward secrecy.** Keys are long-lived RSA. Compromising a private key
  exposes past messages encrypted to it. A production design would use a ratchet
  (e.g. the Signal Double Ratchet / X3DH) for per-message forward secrecy.
- **No public-key authentication (MITM at registration).** The server vouches for
  the mapping `username → public key`. A malicious server could hand out its own
  key. Real systems add safety numbers / key fingerprints users verify
  out-of-band.
- **Metadata is visible.** The server sees who talks to whom and when.
- **Self-signed TLS** for local dev; browsers warn. Use a real CA in production.
- **Key roaming is manual.** The private key is per-browser. Use *Backup my key* to
  export the password-encrypted blob and *Import your key backup* on another device.
  (A server-side encrypted-key vault à la Bitwarden would automate this without
  weakening the model, provided the server never sees the password — i.e. auth via a
  separately derived verifier.)
